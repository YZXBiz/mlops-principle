Machine Learning Operations (MLOps): Overview, Definition, and Architecture#

Authors#

  • Dominik Kreuzberger (KIT, Germany)

  • Niklas Kühl (KIT, Germany)

  • Sebastian Hirschl (IBM, Germany)

Abstract#

The final goal of all industrial machine learning (ML) projects is to develop ML products and rapidly bring them into production. However, it is highly challenging to automate and operationalize ML products and thus many ML endeavors fail to deliver on their expectations. The paradigm of Machine Learning Operations (MLOps) addresses this issue.

This notebook provides a comprehensive overview of MLOps principles, components, roles, and architecture based on a mixed-method research study including literature review, tool review, and expert interviews.

1. Introduction#

Machine Learning (ML) has become an important technique to leverage the potential of data and allows businesses to be more innovative, efficient, and sustainable. However, the success of many productive ML applications in real-world settings falls short of expectations.

Key Challenges#

  • Many ML projects fail to progress from proof of concept to production

  • Data scientists manually manage ML workflows to a great extent

  • Lack of automation and operationalization of ML systems

  • Complex ML system components and infrastructure coordination

Research Question#

What is MLOps?

This research aims to:

  1. Identify important principles of MLOps

  2. Carve out functional core components

  3. Highlight necessary roles for successful MLOps implementation

  4. Derive a general architecture for ML systems design

2. Foundations of DevOps#

DevOps emerged in 2008/2009 as a paradigm addressing social and technical issues in software development organizations.

DevOps Goals#

  • Eliminate the gap between development and operations

  • Emphasize collaboration, communication, and knowledge sharing

  • Ensure automation with CI/CD (Continuous Integration/Continuous Delivery/Continuous Deployment)

  • Enable fast, frequent, and reliable releases

  • Continuous testing, quality assurance, monitoring, logging, and feedback loops

DevOps Tools Categories#

  1. Collaboration and knowledge sharing (e.g., Slack, Trello, GitLab wiki)

  2. Source code management (e.g., GitHub, GitLab)

  3. Build process (e.g., Maven)

  4. Continuous integration (e.g., Jenkins, GitLab CI)

  5. Deployment automation (e.g., Kubernetes, Docker)

  6. Monitoring and logging (e.g., Prometheus, Logstash)

3. Research Methodology#

This study employed a mixed-method approach consisting of:

3.1 Literature Review#

  • Systematic review following Webster and Watson methodology

  • Search query: ((“DevOps” OR “CICD” OR “Continuous Integration” OR “Continuous Delivery” OR “Continuous Deployment”) AND “Machine Learning”) OR “MLOps” OR “CD4ML”

  • Databases: Google Scholar, Web of Science, Science Direct, Scopus, AIS eLibrary

  • Results: 1,864 retrieved articles → 194 screened → 27 selected peer-reviewed articles

3.2 Tool Review#

  • Analysis of open-source tools, frameworks, and commercial cloud ML services

  • 11 tools evaluated for technical components understanding

3.3 Interview Study#

  • Semi-structured expert interviews with theoretical sampling

  • 8 interviews with ML professionals from different organizations, industries, and countries

  • Interviews conducted until theoretical saturation was reached

  • Open coding scheme used for evaluation

4. MLOps Principles#

Based on the research methodology, 9 core principles were identified for realizing MLOps:

P1: CI/CD Automation#

  • Provides continuous integration, continuous delivery, and continuous deployment

  • Carries out build, test, delivery, and deploy steps

  • Provides fast feedback to developers

  • Increases overall productivity

P2: Workflow Orchestration#

  • Coordinates tasks of ML workflow pipeline according to directed acyclic graphs (DAGs)

  • DAGs define task execution order by considering relationships and dependencies

P3: Reproducibility#

  • Ability to reproduce an ML experiment and obtain the exact same results

  • Essential for scientific validity and debugging

P4: Versioning#

  • Ensures versioning of data, model, and code

  • Enables not only reproducibility but also traceability

  • Important for compliance and auditing reasons

P5: Collaboration#

  • Ensures possibility to work collaboratively on data, model, and code

  • Emphasizes collaborative and communicative work culture

  • Aims to reduce domain silos between different roles

P6: Continuous ML Training & Evaluation#

  • Periodic retraining of ML model based on new feature data

  • Enabled through monitoring component, feedback loop, and automated ML workflow pipeline

  • Always includes evaluation run to assess change in model quality

P7: ML Metadata Tracking/Logging#

  • Tracks and logs metadata for each orchestrated ML workflow task

  • Includes training job iteration metadata (date, time, duration)

  • Tracks model-specific metadata (parameters, performance metrics, model lineage)

  • Ensures full traceability of experiment runs

P8: Continuous Monitoring#

  • Periodic assessment of data, model, code, infrastructure resources

  • Monitors model serving performance (e.g., prediction accuracy)

  • Detects potential errors or changes that influence product quality

P9: Feedback Loops#

  • Multiple feedback loops integrate insights from quality assessment into development

  • Feedback from experimental model engineering to feature engineering

  • Feedback from monitoring component to scheduler for retraining enablement

5. Technical Components#

The principles are implemented through 9 technical components:

C1: CI/CD Component (P1, P6, P9)#

  • Ensures continuous integration, delivery, and deployment

  • Handles build, test, delivery, and deploy steps

  • Examples: Jenkins, GitHub Actions

C2: Source Code Repository (P4, P5)#

  • Ensures code storing and versioning

  • Allows multiple developers to commit and merge code

  • Examples: GitHub, GitLab, Bitbucket, Gitea

C3: Workflow Orchestration Component (P2, P3, P6)#

  • Offers task orchestration via directed acyclic graphs (DAGs)

  • Represents execution order and artifact usage of workflow steps

  • Examples: Apache Airflow, Kubeflow Pipelines, Luigi, AWS SageMaker Pipelines

C4: Feature Store System (P3, P4)#

  • Central storage of commonly used features

  • Offline database for experimentation (normal latency)

  • Online database for production predictions (low latency)

  • Examples: Google Feast, Amazon AWS Feature Store, Tecton.ai, Hopswork.ai

C5: Model Training Infrastructure (P6)#

  • Provides foundational computation resources (CPUs, RAM, GPUs)

  • Can be distributed or non-distributed

  • Scalable and distributed infrastructure recommended

  • Examples: Local machines, cloud computation, Kubernetes, Red Hat OpenShift

C6: Model Registry (P3, P4)#

  • Centrally stores trained ML models with metadata

  • Two main functionalities: storing ML artifacts and ML metadata

  • Examples: MLflow, AWS SageMaker Model Registry, Azure ML Model Registry, Neptune.ai

C7: ML Metadata Stores (P4, P7)#

  • Tracks various kinds of metadata for ML workflow pipeline tasks

  • Stores training job metadata (date, time, duration, parameters, performance metrics)

  • Tracks model lineage (data and code used)

  • Examples: Kubeflow Pipelines, AWS SageMaker Pipelines, Azure ML, IBM Watson Studio

C8: Model Serving Component (P1)#

  • Configured for different purposes: online inference or batch inference

  • Can serve via REST API

  • Scalable and distributed infrastructure recommended

  • Examples: Kubernetes + Docker + Flask, KServing, TensorFlow Serving, cloud services

C9: Monitoring Component (P8, P9)#

  • Continuous monitoring of model serving performance

  • Monitors ML infrastructure, CI/CD, and orchestration

  • Examples: Prometheus + Grafana, ELK stack, TensorBoard, cloud monitoring services

6. MLOps Roles#

MLOps requires 7 key roles for successful implementation:

R1: Business Stakeholder#

  • Similar roles: Product Owner, Project Manager

  • Responsibilities: Defines business goals, communicates ROI, presents business value

R2: Solution Architect#

  • Similar roles: IT Architect

  • Responsibilities: Designs architecture, defines technologies after thorough evaluation

R3: Data Scientist#

  • Similar roles: ML Specialist, ML Developer

  • Responsibilities: Translates business problems to ML problems, model engineering, algorithm and hyperparameter selection

R4: Data Engineer#

  • Similar roles: DataOps Engineer

  • Responsibilities: Builds and manages data/feature engineering pipelines, ensures proper data ingestion to feature stores

R5: Software Engineer#

  • Responsibilities: Applies software design patterns, coding guidelines, and best practices to turn raw ML problems into well-engineered products

R6: DevOps Engineer#

  • Responsibilities: Bridges gap between development and operations, ensures CI/CD automation, ML workflow orchestration, model deployment, and monitoring

R7: ML Engineer/MLOps Engineer#

  • Cross-domain role combining aspects of multiple roles

  • Skills from: Data scientists, data engineers, software engineers, DevOps engineers, backend engineers

  • Responsibilities: Builds and operates ML infrastructure, manages automated ML workflow pipelines, handles model deployment and monitoring

7. MLOps Architecture and Workflow#

The end-to-end MLOps architecture consists of 4 main stages:

(A) MLOps Project Initiation#

  1. Business analysis (R1): Identify potential business problems solvable with ML

  2. Architecture design (R2): Define overall ML system architecture and technologies

  3. ML problem derivation (R3): Derive ML problem from business goal

  4. Data requirements (R3, R4): Understand required data to solve the problem

  5. Data source location (R3, R4): Locate and analyze raw data sources

(B) Feature Engineering Pipeline#

B1: Requirements Definition#

  1. Data transformation rules (R4): Define normalization, aggregations, cleaning rules

  2. Feature engineering rules (R3, R4): Define calculation of new features

B2: Pipeline Implementation#

  1. Raw data connection: Connect to streaming, batch, or cloud storage data

  2. Data extraction: Extract data from sources

  3. Data preprocessing: Transform and clean data

  4. Feature engineering: Calculate new features based on existing ones

  5. Data ingestion: Load data into feature store (offline/online databases)

© Experimentation#

  1. Data analysis (R3): Connect to feature store for analysis

  2. Data preparation (R3): Prepare and validate data, create train/test splits

  3. Algorithm selection (R3): Estimate best-performing algorithm and hyperparameters

  4. Model training (R3, R5): Train and validate models iteratively

  5. Model export (R3): Export best-performing model and commit code

(D) Automated ML Workflow Pipeline#

  1. Data extraction: Automated pulling of versioned features

  2. Data preparation: Automated data preparation and validation

  3. Model training: Automated final model training on new data

  4. Model evaluation: Automated model evaluation and hyperparameter adjustment

  5. Model export: Export trained model

  6. Model registry: Push model to registry with metadata

  7. Deployment: CI/CD triggers continuous deployment pipeline

  8. Model serving: Serve predictions on new data

  9. Monitoring: Continuous monitoring of model and infrastructure performance

  10. Feedback loops: Fast feedback for continuous improvement and retraining

  11. Continuous training: Trigger retraining based on monitoring feedback

8. MLOps Definition#

Based on the research findings, MLOps is defined as:

MLOps (Machine Learning Operations) is a paradigm, including aspects like best practices, sets of concepts, as well as a development culture when it comes to the end-to-end conceptualization, implementation, monitoring, deployment, and scalability of machine learning products.

Most of all, it is an engineering practice that leverages three contributing disciplines: machine learning, software engineering (especially DevOps), and data engineering.

MLOps is aimed at productionizing machine learning systems by bridging the gap between development (Dev) and operations (Ops).

Essentially, MLOps aims to facilitate the creation of machine learning products by leveraging these principles: CI/CD automation, workflow orchestration, reproducibility; versioning of data, model, and code; collaboration; continuous ML training and evaluation; ML metadata tracking and logging; continuous monitoring; and feedback loops.

Key Characteristics#

  • Paradigm: More than just tools and technologies

  • Interdisciplinary: Combines ML, software engineering, and data engineering

  • Engineering practice: Focus on production-ready systems

  • Bridge: Connects development and operations

  • Principle-driven: Based on 9 core principles for automation and operationalization

9. Open Challenges#

Several challenges for adopting MLOps have been identified:

9.1 Organizational Challenges#

  • Mindset and culture shift: From model-driven to product-oriented discipline

  • Skills shortage: Lack of highly skilled experts, especially architects, data engineers, ML engineers, DevOps engineers

  • Education gap: MLOps typically not part of data science education

  • Team coordination: Need for multi-disciplinary teams vs. working in silos

  • Communication barriers: Different knowledge levels and specialized terminologies

  • Business buy-in: Convincing decision-makers of MLOps maturity benefits

9.2 ML System Challenges#

  • Fluctuating demand: Designing for varying ML training processes

  • Resource estimation: Difficulty estimating infrastructure resources (CPU, RAM, GPU)

  • Scalability requirements: High level of flexibility needed for infrastructure scaling

9.3 Operational Challenges#

  • Manual operations complexity: Challenging to operate ML manually due to diverse software/hardware stacks

  • Automation requirements: Need for robust automation across the entire pipeline

  • Continuous retraining: Constant incoming data forces repetitive retraining tasks

  • Artifact management: Large number of artifacts require strong governance and versioning

  • Support complexity: Resolving issues complicated by many parties and components involved

  • Failure diagnosis: Failures can be combination of ML infrastructure and software issues

11. Conclusion#

This comprehensive study provides a holistic understanding of MLOps through:

Key Contributions#

  1. 9 Core Principles that guide MLOps implementation

  2. 9 Technical Components that operationalize the principles

  3. 7 Essential Roles required for successful MLOps adoption

  4. End-to-end Architecture with detailed workflow stages

  5. Formal Definition of MLOps as a paradigm

  6. Open Challenges categorized into organizational, system, and operational

Research Impact#

  • Provides common understanding of MLOps terminology and concepts

  • Offers practical guidance for ML researchers and practitioners

  • Addresses the gap between ML model development and production deployment

  • Supports transition from manual ML workflows to automated, operationalized systems

Future Directions#

  • Address identified challenges through targeted solutions

  • Develop MLOps education curricula

  • Create standardized MLOps maturity models

  • Investigate domain-specific MLOps adaptations

  • Explore automation techniques for complex ML workflows

The paradigm of MLOps represents a fundamental shift toward engineering practices that enable reliable, scalable, and maintainable machine learning systems in production environments.
